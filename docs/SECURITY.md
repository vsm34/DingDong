## Pre-Demo Security Audit

Run these commands and verify results before any public demo:

```bash
# Verify google-services.json was never committed
git log --all -- "*/google-services.json"
# Expected: no output

# Verify .env was never committed
git log --all -- ".env"
# Expected: no output

# Verify serviceAccount was never committed
git log --all -- "serviceAccount*"
# Expected: no output

# Scan for potential secrets in history
git log --all -p | grep -i "secret\|password\|token\|key" | grep -v "// \|# \|* "
# Review any matches carefully
```

Test security endpoints:

```bash
# Unauthorized access — must return 401
curl http://dingdong-<id>.local/api/v1/clips

# Wrong token — must return 401
curl -H "Authorization: Bearer wrongtoken" http://dingdong-<id>.local/api/v1/clips

# HMAC missing — Cloud Function must return 401
curl -X POST https://us-central1-dingdong-596c2.cloudfunctions.net/notify \
  -H "Content-Type: application/json" \
  -d '{"deviceId":"test","type":"motion","ts":1234567890}'

# Replayed nonce — must return 401 (send same request twice)
# Expired timestamp — must return 401 (set X-Timestamp to old value)
```

## What Is and Is NOT a Secret

### NOT secret (safe to commit)
- `google-services.json` — Firebase client config, not a server key. However keep out of public repos.
- Firebase project ID (`dingdong-596c2`) — visible in URLs
- Cloud Function URL — public endpoint
- `.env.example` — contains only placeholder values

### IS secret (never commit, never log)
- `device_secret` — HMAC signing key shared between ESP32 and Cloud Function
- Firebase service account JSON — admin access to your Firebase project
- Any Bearer token values — device API access
- Wi-Fi passwords stored in NVS (stay on device, never transmitted after provisioning)

---

## Secrets Storage by Location

| Secret | Where Stored | How Set |
|---|---|---|
| `device_secret` | Firebase Secret Manager + ESP32 NVS | Cloud Function generates, provisions to device |
| Firebase service account | CI/CD env only | Never committed |
| Device Bearer token | ESP32 NVS + app flutter_secure_storage | Generated by ESP32 at provisioning |
| Wi-Fi password | ESP32 NVS only | User enters during onboarding, not stored in cloud |
| Claude/OpenAI API key (Phase 9) | Firebase Secret Manager | Set via Firebase CLI |

## Firestore Security Rules Summary

- Users can only read/write their own `users/{uid}` document
- Devices readable only if `deviceMembers/{deviceId}_{uid}` record exists for the requesting user
- DeviceMembers readable only by the user whose uid matches the record
- Events readable only if user is a member of that device
- **All writes go through Cloud Functions (Admin SDK) only — client cannot write**

Full rules in `cloud/firestore.rules`.

## ESP32 Device Security

- Bearer token generated with `esp_fill_random` (hardware RNG)
- Token comparison via `mbedtls_ct_memcmp` (constant-time, timing-attack resistant)
- Rate limiting: 5 failed auth attempts → 429 for 60 seconds
- `/provision/secret` can only be called once (NVS flag `secret_provisioned`)
- `/provision/status` token returned only once (NVS flag `token_served`)
- HMAC replay protection: nonce + timestamp window ±60s
- Root CA pinned for HTTPS calls to Cloud Function (`*.cloudfunctions.net`)
- Max payload sizes enforced: 2KB settings, 512B provision
